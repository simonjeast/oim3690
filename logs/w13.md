# Week 13 - Final Project Work

## Project

Final project idea: a Group Expense Splitter built with Next.js.

## What I Don't Know Yet

I still need more confidence choosing the right architecture before coding, especially how to separate data logic from UI logic in a way that stays manageable as features are added. I also want to get better at planning error states, loading states, and edge cases before I start building.


### Project Research and Scope

I researched similar expense-splitting apps to compare common features and user flow patterns. I defined a realistic MVP for my timeline: create a group, add members, enter expenses, and show balances. This helped me narrow the project to a version I can complete well instead of trying to build every feature at once.

### Mapping Prior Skills to This Project

From Week 11, I plan to apply `async`/`await` and `fetch()` patterns when handling data operations and any API-style requests, with clear error handling for failed responses. From Week 12, I plan to apply stronger UI-state thinking by showing clear loading, success, and failure states so users always know what is happening.

### Implementation Plan in Next.js

I outlined how I want to structure the app in Next.js: keep reusable UI in components, keep calculation logic in helper functions, and keep data handling separated from display code. I also planned to start with local storage first, then optionally move to a backend later if time allows.

## What I Learned

- Good project outcomes start with scope and research, not just coding right away.
- Earlier class skills are more useful when I intentionally map them to a new project plan.
- Planning loading and error states early improves both code quality and user experience.
- Breaking the work into MVP first, then enhancements, makes the final project feel achievable.
