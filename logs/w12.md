# Week 12 - Project Work

## What I Don't Know Yet

I want more practice with connecting async API responses to UI updates, especially deciding what the page should show while the data is loading or if the request fails. I also want to get more comfortable transforming raw data into useful app-specific results.

### Fetch and Display

The Geolocation DB response includes fields like `city`, `state`, `latitude`, and `longitude`. I used the coordinate fields to calculate the nearest shuttle stop and displayed the API status on the page. The trickiest part was making the app useful even before the API request finished.

### Interactivity and Design

The controls update the schedule results whenever the user changes the origin, destination, or service day. I used a responsive CSS grid so the app feels like a compact transit dashboard on desktop and a single-column tool on mobile.

### Polish and Extend

I added saved preferences with `localStorage`, route browsing, nearest-stop detection, error handling, Sunday/no-route states, and support for the later Thursday/Friday schedule rows.
